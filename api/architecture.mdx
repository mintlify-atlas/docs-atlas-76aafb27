## API Routes Architecture

### REST API Structure

All API routes follow Next.js App Router conventions:

```typescript
// Example: /app/api/servers/route.ts
export const GET = async (req: Request) => { /* ... */ }
export const POST = async (req: Request) => { /* ... */ }

// Example: /app/api/servers/[serverId]/route.ts
export const GET = async (
  req: Request,
  { params }: { params: { serverId: string } }
) => { /* ... */ }
```

### Authentication Flow

```typescript
import { currentProfile } from "@/lib/current-profile";

export const GET = async (req: Request) => {
  // 1. Verify authentication
  const profile = await currentProfile();
  if (!profile) {
    return new Response("Unauthorized", { status: 401 });
  }

  // 2. Perform authorized operation
  const data = await db.server.findMany({
    where: {
      members: {
        some: { profileId: profile.id },
      },
    },
  });

  // 3. Return response
  return NextResponse.json(data);
};
```

## Real-time Layer

### WebSocket Server

The WebSocket server runs on port 1234 alongside Next.js:

```json
{
  "dev": "concurrently \"next\" \"set HOST=127.0.0.1&& set PORT=1234 && npx y-websocket\" --kill-others"
}
```

Source: `package.json:6`

### Yjs Integration

Yjs provides CRDT-based synchronization:

```typescript
const yDoc = new Y.Doc({ gc: true });
const yStore = new YKeyValue(
  yDoc.getArray<{ key: string; val: TLRecord }>(`tl_${roomId}`)
);

const room = new WebsocketProvider(
  hostUrl,
  roomId,
  yDoc,
  { connect: true }
);
```

Source: `/useYjsStore.ts:47`

### Data Flow

1. **User Action** → tldraw store
2. **Store Change** → Yjs document (via listener)
3. **Yjs Document** → WebSocket server
4. **WebSocket Server** → Other clients
5. **Other Clients** → Update their tldraw stores

See [Real-time Sync](/api/real-time-sync) for detailed documentation.

## Authentication System

### Clerk Integration

TeamTalker uses Clerk for authentication:

```typescript
import { auth } from "@clerk/nextjs";
import { db } from "@/lib/db";

export const currentProfile = async () => {
  const { userId } = auth();

  if (!userId) {
    return null;
  }

  const profile = await db.profile.findUnique({
    where: { userId },
  });

  return profile;
};
```

Source: `/lib/current-profile.ts:4`

### Authentication Flow

1. User signs in via Clerk UI
2. Clerk creates session
3. API routes call `currentProfile()`
4. Profile is fetched from database using `userId`
5. Authorization checks are performed

## File Upload System

### UploadThing Integration

TeamTalker uses UploadThing for image uploads:

```typescript
// app/api/uploadthing/core.ts
import { createUploadthing } from "uploadthing/next";

const f = createUploadthing();

export const ourFileRouter = {
  serverImage: f({ image: { maxFileSize: "4MB" } })
    .middleware(async ({ req }) => {
      const profile = await currentProfile();
      if (!profile) throw new Error("Unauthorized");
      return { userId: profile.userId };
    })
    .onUploadComplete(async ({ metadata, file }) => {
      console.log("Upload complete for userId:", metadata.userId);
      console.log("file url", file.url);
    }),
};
```

Used for:
- Server images/icons
- User profile pictures

## Security Considerations

### Authentication

- All API routes verify authentication via Clerk
- Profile lookup prevents unauthorized access
- Session management handled by Clerk

### Authorization

- Server ownership checks (`profileId` matching)
- Role-based permissions (ADMIN, MODERATOR, GUEST)
- Cannot modify own role or remove self (server owner)

### Database

- Cascade deletes prevent orphaned records
- Unique constraints on `userId` and `inviteCode`
- Indexes for performance on foreign keys

---

## Performance Optimizations

### Database

- Indexes on `profileId` and `serverId` for faster queries
- Efficient Prisma queries with `include` for joins
- Connection pooling via Prisma

### Real-time

- Yjs garbage collection to prevent memory bloat
- `requestAnimationFrame` for batched presence updates
- Local-only changes ignored in Yjs listeners

### Frontend

- Next.js automatic code splitting
- React 18 concurrent rendering
- Image optimization via Next.js Image component
